## 3.0. Вклад автора в контексте главы

Настоящая глава содержит два типа результатов.

**Результаты, заимствованные из первичных источников** и используемые
как теоретическая основа:

- Математический аппарат конечных структур Крипке и параллельной
  композиции процессов (von Hippel (2025), p. 62–65).
- Формальные описания механизмов: Algorithm 1, formulas (3.1)–(3.11),
  Observations 1–6, Theorems 1–4 (von Hippel (2025), p. 17–49).
- Формулы вычисления RTO по RFC 6298 (Paxson et al., 2011).

**Оригинальные результаты, полученные автором:**

1. **Три JSON-модели** K_Karn, K_RTO, K_GBN — машиночитаемые описания
   механизмов в едином формате, пригодном для автоматической верификации.

2. **CTL-спецификации** для каждой модели (22 спецификации суммарно) —
   переформулировка теоретических результатов von Hippel (доказанных в
   Ivy и ACL2s) в язык CTL для верификации в nuXmv. Von Hippel явно
   указывает, что соединение моделей разных уровней протокола является
   открытой задачей (p. 97); настоящая работа реализует такое соединение.

3. **Генератор SMV** (`kripke_generator`) — инструмент трансляции JSON
   в формат nuXmv с поддержкой полей `specifications` и `fairness`.

4. **Итерационное уточнение моделей** (6 итераций суммарно) — анализ
   контрпримеров nuXmv и уточнение спецификаций, выявившее семантически
   содержательные граничные случаи в каждом из трёх механизмов.

5. **Результаты верификации** — 22 доказанных CTL-свойства, покрывающих
   свойства безопасности и живости трёх механизмов планирования канальных
   ресурсов.

6. **Анализ вертикальной цепочки** K_Karn → K_RTO → K_GBN как единой
   методологии верификации механизмов планирования канальных ресурсов.


## 3.1. Введение

Планирование канальных ресурсов в транспортных протоколах представляет
собой совокупность взаимосвязанных механизмов, совместно определяющих,
как и когда отправитель использует пропускную способность сетевого канала.
К таким механизмам относятся: алгоритм измерения круговой задержки (RTT),
вычисление таймаута ретрансмиссии (RTO) и протокол скользящего окна
Go-Back-N, лежащий в основе механизма TCP. Корректность каждого из
них критична для стабильности всей сети: RFC 6298 прямо указывает, что
*«The Internet, to a considerable degree, relies on the correct implementation
of the RTO algorithm in order to preserve network stability and avoid
congestion collapse»* (von Hippel (2025), p. 11).

Несмотря на широкое практическое применение этих механизмов, до работы
von Hippel (2025) ни один из них не был формально верифицирован. Настоящая
глава восполняет этот пробел применительно к теме планирования канальных
ресурсов: для каждого из трёх механизмов построена формальная модель
Крипке, сформулированы верифицируемые свойства и получены результаты
автоматической верификации инструментом nuXmv 2.1.0.

## 3.2. Методологическая основа

### 3.2.1. Конечные структуры Крипке

Формальной основой настоящей главы служит аппарат **конечных структур
Крипке** и **темпоральной логики CTL**, описанный в разделе 4.3 диссертации
von Hippel (2025), стр. 62–65.

**Определение 1 (Конечная структура Крипке, von Hippel (2025), p. 62).**
Конечная структура Крипке — это кортеж $K = \langle AP, S, s_0, T, L \rangle$,
где:

- $AP$ — конечное множество атомарных пропозиций;
- $S$ — конечное множество состояний;
- $s_0 \in S$ — начальное состояние;
- $T \subseteq S \times S$ — отношение переходов;
- $L : S \to 2^{AP}$ — тотальная функция разметки.

Каждый компонент системы формализуется как **Процесс** — структура Крипке,
дополненная маркировкой переходов входными и выходными событиями
(Definition 2, von Hippel (2025), p. 63):

$$P = \langle AP, I, O, S, s_0, T, L \rangle, \quad T \subseteq S \times (I \cup O) \times S$$

Глобальная модель системы строится через **параллельную композицию**
процессов с синхронизацией на общих событиях (Definition 3, формула (4.3),
von Hippel (2025), p. 64):

$$P_1 \parallel P_2 = \langle AP_1 \cup AP_2,\;
(I_1 \cup I_2) \setminus (O_1 \cup O_2),\;
O_1 \cup O_2,\; S_1 \times S_2,\; (s_0^1, s_0^2),\; T, L \rangle$$

Проекция параллельной композиции на конечную структуру Крипке (удаление
меток событий с переходов, добавление функции $L$) даёт модель, пригодную
для верификации с помощью model checker.

### 3.2.2. Язык спецификаций CTL

Свойства моделей выражаются в **вычислительной темпоральной логике CTL**,
расширяющей LTL кванторами по путям. Основные операторы:

```
AG φ  — «во всех состояниях всех путей φ истинна» (safety)
AF φ  — «на всех путях φ рано или поздно наступит» (liveness)
EX φ  — «существует следующее состояние, в котором φ истинна»
```

Верификация выполняется инструментом **nuXmv 2.1.0** методом проверки
моделей на основе бинарных диаграмм решений (BDD). При обнаружении
нарушения спецификации nuXmv автоматически строит **контрпример** —
конкретную трассу состояний, на которой свойство нарушается. Анализ
контрпримеров является неотъемлемой частью методологии и позволяет
итеративно уточнять как модели, так и спецификации.

### 3.2.3. Конечная абстракция

Пространства состояний рассматриваемых механизмов в общем случае
бесконечны: номера пакетов, значения таймеров и числовые переменные
алгоритмов принимают значения из счётных множеств. Для построения конечных
структур Крипке применяется **качественная абстракция** — каждое состояние
кодирует фазу работы алгоритма или протокола, а не конкретные числовые
значения. Этот подход является стандартным в верификации сетевых протоколов
методом model checking (Baier, Katoen, 2008) и используется, в частности,
в работе von Hippel (2025) при построении моделей рукопожатий TCP, DCCP и
SCTP (глава 4, стр. 60–78).

## 3.3. Три механизма как единая цепочка

Ключевая идея настоящей главы состоит в том, что три рассматриваемых
механизма образуют **вертикальную цепочку зависимостей**, которая в
совокупности реализует планирование канальных ресурсов:

```
┌─────────────────────────────────────────────────────────────┐
│              Цепочка механизмов планирования                 │
│                                                             │
│   snds(i)/rcvs(j)         S (выборки RTT)                  │
│   ───────────────►  K_Karn ──────────────►  K_RTO          │
│   наблюдение за             алгоритм        вычисление RTO  │
│   отправителем              Карна           по RFC 6298     │
│                                    │                        │
│                                    │ rto_adequate /         │
│                                    │ spurious_timeout        │
│                                    ▼                        │
│                                 K_GBN                       │
│                              протокол GB(N)                  │
│                              (основа TCP sliding window)    │
└─────────────────────────────────────────────────────────────┘
```

**K_Karn** (раздел 3.X) — немешающий монитор над исполнением отправителя
(von Hippel (2025), p. 17). Наблюдает за событиями `snds(i)` и `rcvs(j)`,
принимает решение о допустимости выборки RTT согласно условию ok-to-sample
(p. 20, Algorithm 1, p. 19), выдаёт выборку $S = \tau - time[high]$.
Верифицируемые свойства охватывают Наблюдения 1–4 диссертации (стр. 17–21).

**K_RTO** (раздел 3.Y) — вычислительный механизм, использующий выборки
алгоритма Карна как вход. Реализует формулы RFC 6298 (p. 21): вычисляет
сглаженный RTT (`srtt`), дисперсию (`rttvar`) и таймаут ретрансмиссии
(`rto = srtt + max(G, 4·rttvar)`). Верифицируемые свойства охватывают
Наблюдения 5–6 (стр. 22–23) и свойства сходимости оценок.

**K_GBN** (раздел 3.Z) — протокол автоматического запроса повторной передачи,
использующий RTO как таймер окна. Система состоит из четырёх компонентов:
Sender, TBF_s, TBF_r, Receiver, связанных через параллельную композицию
(формула (3.11), p. 46). Верифицируемые свойства охватывают инварианты
Теорем 1–4 и свойства эффективности канала (стр. 39–49).

Именно эта вертикальная цепочка отражает архитектуру планирования канальных
ресурсов: качество оценки RTT определяет точность RTO, точность RTO
определяет эффективность протокола скользящего окна. Von Hippel (2025)
явно обозначает соединение этих компонентов как открытую задачу: *«We did
not connect our finite state handshake models to our infinite state models
of Karn's Algorithm, the RTO computation, or Go-Back-N»* (p. 97). Настоящая
глава реализует именно такое соединение в рамках единой методологии.

## 3.4. Сравнительная характеристика трёх моделей

| Характеристика | K_Karn | K_RTO | K_GBN |
|---|---|---|---|
| **Тип механизма** | Монитор | Вычисление | Протокол |
| **Состояний** | 12 | 15 | 13 |
| **Переходов** | 19 | 22 | 21 |
| **Атомарных пропозиций** | 15 | 19 | 6 |
| **CTL-спецификаций** | 9 | 7 | 6 |
| **Ограничение FAIRNESS** | да | да | нет |
| **Итераций уточнения** | 2 | 3 | 1 |
| **Наблюдения/теоремы** | Obs.1–4 (p. 17–21) | Obs.5–6 (p. 22–23) | Thm.1–4 (p. 39–46) |
| **Верификатор** | nuXmv 2.1.0 | nuXmv 2.1.0 | nuXmv 2.1.0 |
| **Результат** | 9/9 `is true` | 7/7 `is true` | 6/6 `is true` |

### Роль ограничений справедливости

K_Karn и K_RTO потребовали введения ограничений справедливости `FAIRNESS`,
тогда как K_GBN — нет. Это отражает принципиальное различие в природе
механизмов:

- **K_Karn и K_RTO** содержат потенциально бесконечные ретрансмиссионные
  петли (бесконечные потери ACK), которые технически допустимы в
  недетерминированном автомате, но семантически нереалистичны. Ограничение
  справедливости формализует факт ограниченного числа ретрансмиссий в
  реальных протоколах (`tcp_retries_max`).

- **K_GBN** таких петель не содержит: механизм `timeout(s): curPkt ← hiAck`
  гарантирует выход из любого состояния заполненного окна через явный
  откат, не требующий ограничения справедливости.

### Итерационный характер верификации

Все три модели потребовали итеративного уточнения спецификаций после
анализа контрпримеров nuXmv. Это не является недостатком подхода —
напротив, именно в этом состоит ценность model checking: автоматически
найденные контрпримеры выявляют семантические неточности в первоначальных
формулировках свойств. Сводка итераций:

| Модель | Спецификация | Причина уточнения |
|---|---|---|
| K_GBN | φ₃: `window_full → AF transmitting` | Не учтено терминальное состояние `Connection_complete` |
| K_RTO | φ₃: `sampling_blocked → AF sampling_enabled` | Не учтено завершение (`Transmission_complete`) и разрыв (`Connection_aborted`) |
| K_RTO | φ₃ (2-я итерация) | Бесконечная ретрансмиссия — введение `FAIRNESS` |
| K_Karn | φ₇: `Sample_produced → ACK_fresh` | Предикат `ACK_fresh` не наследовался в s8/s9 |
| K_Karn | φ₈: `Window_ambiguous → ...` | Устаревший ACK — законный исход, добавлен `ACK_stale` |

## 3.5. Инструментальное обеспечение

Верификация выполнялась с использованием следующего инструментального стека:

**Формат модели:** JSON. Каждая модель описывается кортежем
`{states, initial_states, transitions, state_predicates, specifications,
fairness}`. Формат совместим с моделями из бакалаврской работы автора
(протокол СТП-ИСС-1), что обеспечивает преемственность инструментария.

**Генератор SMV:** программа на C++ (`kripke_generator`), транслирующая
JSON-описание модели в формат SMV, пригодный для верификации инструментом
nuXmv. Генератор поддерживает поля `specifications` (CTL-спецификации)
и `fairness` (ограничения справедливости).

**Верификатор:** nuXmv 2.1.0 (Fondazione Bruno Kessler), метод BDD.
При нарушении спецификации автоматически строится трасса-контрпример.

```
Пайплайн верификации:

  K_XXX.json
      │
      ▼
  kripke_generator   →   K_XXX_output.smv
  (C++)
      │
      ▼
  nuXmv 2.1.0        →   результат верификации
                          (true / false + контрпример)
```

## 3.6. Структура главы

Глава организована следующим образом.

**Раздел 3.X. Модель K_GBN** — формальная модель Крипке протокола Go-Back-N.
Описывает систему из четырёх компонентов (Sender, TBF_s, TBF_r, Receiver),
их параллельную композицию и CTL-верификацию свойств корректности и
эффективности использования канала. Теоретическая основа: глава 3 диссертации
von Hippel (2025), стр. 30–59.

**Раздел 3.Y. Модель K_RTO** — формальная модель Крипке механизма вычисления
таймаута ретрансмиссии по алгоритму Карна и RFC 6298. Верифицирует свойства
сходимости оценок RTT и адекватности RTO. Теоретическая основа: глава 2
диссертации von Hippel (2025), стр. 11–29; RFC 6298 (Paxson et al., 2011).

**Раздел 3.Z. Модель K_Karn** — формальная модель Крипке алгоритма Карна
как немешающего монитора над исполнением отправителя. Верифицирует
Наблюдения 1–4 о корректности выборки RTT. Теоретическая основа: разделы
2.2–2.4 диссертации von Hippel (2025), стр. 14–21; Karn & Partridge (1987).
